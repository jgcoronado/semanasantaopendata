## Critical Issues (Must Fix)

### Issue 1: No skip-navigation link

**WCAG Criterion:** 2.4.1 — Bypass Blocks (Level A)
**Severity:** Critical
**Location:** `src/layouts/Base.astro:52`
**Element:** `<header class="cabecera">` — no skip link precedes it

**Problem:**
Every page starts with a navigation bar containing year links (variable number), "Hermandades", and "Comparativa". There is no mechanism to bypass this repeated block.

**Impact:**
Keyboard-only users and screen reader users must tab through every navigation link on every page before reaching page content. On a year with 7 days plus 2 static links that is 9 Tab presses minimum on every page load.

**Remediation:**
Add a visually-hidden skip link as the very first focusable element inside `<body>`. Make it visible on focus.

```astro
<!-- Base.astro — add immediately after <body> -->
<a href="#main-content" class="saltar-contenido">Saltar al contenido</a>
<!-- ... existing header ... -->
<main id="main-content">
  <slot />
</main>
```

```css
/* global.css */
.saltar-contenido {
  position: absolute;
  top: -100%;
  left: 1rem;
  background: var(--morado);
  color: var(--blanco);
  padding: 0.5rem 1rem;
  border-radius: 0 0 var(--radio) var(--radio);
  font-weight: 700;
  text-decoration: none;
  z-index: 9999;
}
.saltar-contenido:focus {
  top: 0;
}
```

**Testing:**
- Manual: Press Tab on any page — first focus should land on "Saltar al contenido", pressing Enter should move focus to `#main-content`.
- Automated: axe rule `bypass`.

### Issue 2: Canvas charts have no accessible data alternative

**WCAG Criterion:** 1.1.1 — Non-text Content (Level A)
**Severity:** Critical
**Location:**
- `src/components/Explorador.astro:41` (`grafica-dias`)
- `src/components/HorariosGraficas.astro:50,57` (gantt, ocupacion)
- `src/components/GraficasAnalisis.astro:88,116,139,157,163` (5 canvases)
- `src/components/CarreraOficialGraficas.astro:51,73,78,100,129,135` (6 canvases)
- `src/components/GraficasHermandad.astro:46,54,61` (per-year pies, histogram, composition)
- `src/components/ComposicionDia.astro:24,28` (2 bar charts)

**Problem:**
All charts are rendered as `<canvas>` elements. The `aria-label` attribute provides a title string (e.g., `"Nazarenos y cortejo total por día"`) but the actual data is inaccessible — a screen reader announces only that string with no values. Chart.js renders exclusively to canvas; no text alternative is inside the element.

**Impact:**
Blind users and users with visual impairments relying on screen readers receive no data from any chart. The data exists (in JSON blobs and, for the nazarenos table, in the sortable table), but charts like the Gantt timeline, scatter plots, box plots, and carrera acumulado charts have no textual equivalent at all.

**Remediation:**
Two complementary approaches — apply both:

**A. Add a meaningful `<p>` summary inside each `<figure>` (quick win):**

```astro
<!-- Example for Explorador.astro -->
<figure class="grafica grafica--ancha">
  <figcaption>Nazarenos y cortejo total por día</figcaption>
  <div class="grafica__lienzo"><canvas id="grafica-dias" aria-hidden="true"></canvas></div>
  <p class="grafica__nota visualmente-oculto" id="grafica-dias-desc">
    Gráfica de barras con los totales de nazarenos y del cortejo por cada día procesional.
    Los datos completos están disponibles en la tabla de esta página.
  </p>
</figure>
```

Switch the `aria-label` on `<canvas>` to `aria-hidden="true"` and let the `<figure>/<figcaption>` be the accessible label. Use `aria-describedby` to link the description:

```html
<figure class="grafica grafica--ancha" aria-describedby="grafica-dias-desc">
```

**B. For data-heavy charts (Gantt, acumulado, dispersión), add a visually-hidden summary table:**

Since many charts duplicate data already in sortable tables on the page, a note pointing to the table is sufficient:

```astro
<p class="visualmente-oculto">
  Estos datos están disponibles en la tabla «Horarios oficiales» más arriba en esta página.
</p>
```

**Testing:**
- Manual: Navigate to each chart with a screen reader — confirm the announced text is meaningful.
- Automated: axe rule `canvas-img-alternative`.

### Issue 3: Radio groups lack programmatic grouping

**WCAG Criterion:** 1.3.1 — Info and Relationships (Level A)
**Severity:** Critical
**Location:**
- `src/pages/comparativa.astro:50-58` (Métrica segmento)
- `src/components/GraficasHermandad.astro:32-39` (Métrica de las tartas)

**Problem:**
The segmented radio-button controls use `<span class="campo">` with adjacent text ("Métrica") and `<span class="segmento">` containing `<label>/<input>` pairs. The group label ("Métrica") is not programmatically associated with the radio group — it is just visible text in a `<span>`.

```html
<!-- Current (broken grouping) -->
<span class="campo" style="gap:.4rem">
  Métrica
  <span class="segmento">
    <label><input type="radio" name="metrica" value="noNaz" checked /> Nazarenos</label>
    <label><input type="radio" name="metrica" value="noTotal" /> Cortejo total</label>
  </span>
</span>
```

Screen readers announce each radio as "Nazarenos, radio button" without the group label "Métrica".

**Impact:**
Screen reader users don't know what metric they are selecting. Particularly confusing on comparativa.astro where two separate control groups are present.

**Remediation:**

```html
<!-- Option A: fieldset/legend (most compatible) -->
<fieldset class="campo segmento-group">
  <legend>Métrica</legend>
  <span class="segmento">
    <label><input type="radio" name="metrica" value="noNaz" checked /> Nazarenos</label>
    <label><input type="radio" name="metrica" value="noTotal" /> Cortejo total</label>
  </span>
</fieldset>
```

Add CSS to strip the fieldset border/padding to maintain visual design:

```css
.segmento-group {
  border: none;
  padding: 0;
  margin: 0;
}
.segmento-group legend {
  font-size: 0.85rem;
  color: var(--gris);
  margin-bottom: 0.25rem;
  padding: 0;
}
```

Note: GraficasHermandad.astro already has `role="group" aria-label="Métrica de las tartas"` on the outer div — that pattern is acceptable. Apply it consistently to comparativa.astro too.

**Testing:**
- Manual: Tab to a radio button with a screen reader — confirm it announces "Métrica, Nazarenos, radio button, 1 of 2".

### Issue 5: FuenteBadge small-text contrast may fail for 'oficial', 'prensa', 'propia' types

**WCAG Criterion:** 1.4.3 — Contrast (Minimum) (Level AA)
**Severity:** Serious
**Location:** `src/components/FuenteBadge.astro:65-70`
**Element:** `.fuente-badge` text at `0.78rem`

**Problem:**
FuenteBadge uses `color-mix(in srgb, var(--fuente-color) 85%, #000)` for text on a `color-mix(in srgb, var(--fuente-color) 8%, transparent)` background. For types with mid-luminance source colors the resulting contrast can fall below 4.5:1:

| Type      | Source color | Approx. text | Approx. bg  | Est. contrast |
|-----------|-------------|-------------|-------------|---------------|
| oficial   | `#1192e8`   | `#0e7cc5`   | `#ecf6fd`   | ~4.37:1 ✗    |
| prensa    | `#fa4d56`   | `#d54149`   | `#fff1f2`   | ~4.08:1 ✗    |
| propia    | `#b28600`   | `#977200`   | `#f9f5eb`   | ~4.16:1 ✗    |
| hermandad | `#198038`   | `#156d30`   | `#ecf5ef`   | ~5.82:1 ✓    |

Note: `color-mix` output is browser-dependent; verify with DevTools.

**Remediation:**
Darken the text mix more aggressively (e.g. 70% source + 30% black) or use a fixed sufficiently-dark text color per type:

```css
/* FuenteBadge.astro <style> */
.fuente-badge {
  color: color-mix(in srgb, var(--fuente-color) 65%, #000); /* was 85% */
}
```

Or define per-type text colors in the frontmatter:

```js
const textColorTipo = {
  oficial:   '#0a5fa0',  /* verified contrast on its badge bg */
  prensa:    '#b5383f',
  hermandad: '#156d30',
  propia:    '#7a5c00',
};
```

### Issue 6: No visible focus style on sort-column buttons

**WCAG Criterion:** 2.4.7 — Focus Visible (Level AA)
**Severity:** Serious
**Location:** `src/styles/global.css:171-174`
**Element:** `table.datos th.orden button`

**Problem:**
The sort buttons in `TablaNazarenos`, `HorariosTabla`, and `CarreraOficialTabla` have their default browser outline suppressed by the inherited `* { box-sizing: border-box }` block and the button reset CSS (`background: none; border: 0`). No `:focus-visible` style is declared. Some browsers show an outline but it may be invisible against `var(--fondo-alt)`.

**Remediation:**

```css
/* global.css — add after the sort button hover rule */
table.datos th.orden button:focus-visible {
  outline: 2px solid var(--morado);
  outline-offset: 2px;
  border-radius: 4px;
}
```

### Issue 7: Segmented radio inputs show no focus ring

**WCAG Criterion:** 2.4.7 — Focus Visible (Level AA)
**Severity:** Serious
**Location:** `src/styles/global.css:232-235`
**Element:** `.segmento label` / `.segmento input`

**Problem:**
The `.segmento` control visually hides the radio `<input>` behind the label (the input is inline but the checked-label background overrides focus appearance). No `:focus-visible` rule on `.segmento label` exists.

**Remediation:**

```css
/* global.css */
.segmento label:has(input:focus-visible) {
  outline: 2px solid var(--morado);
  outline-offset: 1px;
}
```

### Issue 10: `prefers-reduced-motion` not respected

**WCAG Criterion:** 2.3.3 — Animation from Interactions (Level AAA, best practice for AA)
**Severity:** Moderate
**Location:** `src/styles/global.css` (transitions on `.dia-card`, `.anio-card`, `.btn`, etc.)

**Problem:**
Several elements use CSS `transition: transform` and `transform: translateY(-2px)` on hover. Chart.js uses animations by default. Users with vestibular disorders or epilepsy may request reduced motion via OS settings.

**Remediation:**

```css
/* global.css — add at the end */
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    transition-duration: 0.01ms !important;
    animation-duration: 0.01ms !important;
  }
}
```

For Chart.js, pass `animation: false` in each chart's options object when the media query matches:

```js
const reducedMotion = window.matchMedia('(prefers-reduced-motion: reduce)').matches;
// In each new Chart(..., { options: { animation: reducedMotion ? false : { ... } } })
```

### Issue 11: KPI cards have no semantic role / accessible name

**WCAG Criterion:** 1.3.1 — Info and Relationships (Level A)
**Severity:** Moderate
**Location:** `src/pages/[anio]/hermandad/[slug].astro:149-165`
**Element:** `<div class="kpi">`

**Problem:**
KPI cards use `<div class="kpi__num">` and `<div class="kpi__lbl">` without any semantic association between the number and its label. Screen readers read the number and label as separate pieces of text in sequence, but there is no programmatic `<dl>/<dt>/<dd>` relationship.

**Remediation:**
Replace with a `<dl>` definition list:

```html
<dl class="kpis">
  <div class="kpi">
    <dt class="kpi__lbl">Nazarenos</dt>
    <dd class="kpi__num">{fmt(r.noNaz)}</dd>
  </div>
  ...
</dl>
```

Or add `aria-labelledby` pairing — though the `<dl>` approach is simpler and semantically correct.

### Issue 13: `fuente-badge__tipo` text uppercased via CSS only

**WCAG Criterion:** 1.3.1 (Level A)
**Severity:** Minor
**Location:** `src/components/FuenteBadge.astro:86-90`

**Problem:**
`text-transform: uppercase` is applied via CSS to `.fuente-badge__tipo`. Some older screen readers announce the CSS-transformed text in all-caps letter-by-letter (e.g., "O F I C I A L"). Modern screen readers generally handle this correctly, but it's a risk.

**Remediation:**
No code change needed for modern browsers, but if target audience includes older AT: store the `etiqueta` already in the desired casing and remove `text-transform: uppercase` from CSS.

### Issue 14: Charts do not suppress animation on `prefers-reduced-motion` (Chart.js-specific)

**WCAG Criterion:** 2.3.3 (Level AAA / best practice)
**Severity:** Minor
**Location:** All `<script>` blocks initialising `new Chart(...)` across components

**Problem:**
Chart.js runs entrance animations by default. Users with `prefers-reduced-motion` enabled will see charts animate regardless.

**Remediation:**
Add a shared utility or add per-chart:

```js
const noAnim = window.matchMedia('(prefers-reduced-motion: reduce)').matches;
new Chart(canvas, {
  ...
  options: {
    animation: noAnim ? false : undefined,
    ...
  }
});
```
